二、功能模块与实现代码（按接口）
=================================

模块：用户登录与鉴权
1) POST /user/code
- 调用：UserServiceImpl.sendCode(phone, session)
- 代码：src\main\java\com\hmdp\service\impl\UserServiceImpl.java
- 逻辑：校验手机号 -> 生成6位验证码 -> 写Redis并设置2分钟TTL
- Redis：
  key=login:code:{phone}，类型=String，value=验证码，TTL=2分钟
- 数据库：无直接写库

2) POST /user/login
- 调用：UserServiceImpl.login(loginForm, session)
- 逻辑：校验手机号+验证码 -> 查tb_user，不存在则创建用户 -> 生成token -> 用户信息写Redis Hash -> 返回token
- Redis：
  key=login:code:{phone}，类型=String，读取验证码
  key=login:token:{token}，类型=Hash，field为UserDTO字段，TTL=30分钟（拦截器续期）
- 数据库：
  表tb_user（查phone、必要时新增用户）

3) POST /user/logout
- 调用：UserController.logout
- 逻辑：清ThreadLocal -> 删除login:token:{token} -> 失效session
- Redis：
  key=login:token:{token}，类型=Hash，删除
- 数据库：无

4) 鉴权拦截器
- RefreshTokenInterceptor：读取authorization token -> 从Redis Hash取用户 -> 放入UserHolder -> 刷新TTL 30分钟
- LoginInterceptor：校验UserHolder是否有用户，没有则401
- 代码：
  src\main\java\com\hmdp\interceptor\RefreshTokenInterceptor.java
  src\main\java\com\hmdp\interceptor\LoginInterceptor.java
  src\main\java\com\hmdp\config\MvcConfig.java

5) GET /user/me, GET /user/{id}, GET /user/info/{id}
- 逻辑：当前登录用户信息、用户基础信息、用户详情信息
- 数据库：tb_user, tb_user_info
- Redis：依赖登录态login:token:{token}

6) 用户签到
- POST /user/sign -> UserServiceImpl.sign()
- GET /user/sign/count -> UserServiceImpl.signCount()
- 逻辑：按月位图签到，bitField统计连续签到天数
- Redis：
  key=sign:{userId}:yyyyMM，类型=Bitmap(String位操作)
- 数据库：代码未使用tb_sign（当前实现走Redis位图）

-------------------------------------------------
模块：商铺
1) GET /shop/{id}
- 调用：ShopServiceImpl.queryById(id)
- 逻辑：先读Redis缓存；miss则查库并写逻辑过期缓存；命中后走逻辑过期策略（过期则异步重建）
- 代码：
  src\main\java\com\hmdp\service\impl\ShopServiceImpl.java
  src\main\java\com\hmdp\utils\CacheClient.java
- Redis：
  key=cache:shop:{id}，类型=String(JSON，内部含expireTime+data)
  key=lock:shop:{id}，类型=String(互斥锁，缓存重建)
- 数据库：
  表tb_shop

2) PUT /shop
- 调用：ShopServiceImpl.update(shop)
- 逻辑：更新tb_shop后删除cache:shop:{id}实现缓存一致性
- Redis：
  key=cache:shop:{id} 删除
- 数据库：
  表tb_shop

3) GET /shop/of/type?typeId=&current=&x=&y=
- 调用：ShopServiceImpl.queryShopByType(typeId,current,x,y)
- 逻辑：
  有坐标：使用Redis GEO做5km范围按距离分页；若该type GEO无数据先从DB预热
  无坐标：直接DB分页
- Redis：
  key=shop:geo:{typeId}，类型=GEO(ZSet底层)
- 数据库：
  表tb_shop（按type_id查，预热经纬度）

4) GET /shop/of/name
- 逻辑：按名称模糊查店铺
- 数据库：tb_shop
- Redis：无

5) POST /shop
- 逻辑：新增商铺
- 数据库：tb_shop
- Redis：无（后续通过查询触发缓存）

-------------------------------------------------
模块：商铺类型
1) GET /shop-type/list
- 调用：ShopTypeServiceImpl.querySort()
- 逻辑：先读Redis ZSet；没有则查库按sort排序并回填Redis
- Redis：
  key=shop:type:list，类型=ZSet(member=typeId, score=sort)，TTL=1小时
- 数据库：
  表tb_shop_type

-------------------------------------------------
模块：博客/探店笔记
1) POST /blog
- 调用：BlogServiceImpl.saveBlog(blog)
- 逻辑：保存笔记到tb_blog -> 查询粉丝(tb_follow) -> 推送笔记id到每个粉丝收件箱ZSet
- Redis：
  key=feed:{followerUserId}，类型=ZSet(member=blogId,score=时间戳)
- 数据库：
  表tb_blog, tb_follow

2) PUT /blog/like/{id}
- 调用：BlogServiceImpl.updateLike(id)
- 逻辑：基于ZSet判断是否已点赞，已点赞则取消，否则点赞；同步更新tb_blog.liked
- Redis：
  key=blog:liked:{blogId}，类型=ZSet(member=userId, score=点赞时间戳)
- 数据库：
  表tb_blog（liked字段增减）

3) GET /blog/hot
- 调用：BlogServiceImpl.queryHotBlog(current)
- 逻辑：按liked降序查tb_blog，补充作者信息和当前用户点赞状态
- Redis：
  读blog:liked:{blogId}判断isLike
- 数据库：
  表tb_blog, tb_user

4) GET /blog/{id}
- 调用：BlogServiceImpl.queryBlogById(id)
- 逻辑：查笔记详情+作者信息+点赞状态
- Redis：blog:liked:{blogId}
- 数据库：tb_blog, tb_user

5) GET /blog/likes/{id}
- 调用：BlogServiceImpl.queryBlogLikes(id)
- 逻辑：取点赞ZSet top5用户，按顺序查询用户列表
- Redis：
  key=blog:liked:{blogId}，类型=ZSet
- 数据库：
  表tb_user

6) GET /blog/of/follow?lastId=&offset=
- 调用：BlogServiceImpl.queryBlogOfFollow(max,offset)
- 逻辑：滚动分页读收件箱ZSet；无数据时从DB预热关注博主的笔记到Redis
- Redis：
  key=feed:{userId}，类型=ZSet
- 数据库：
  表tb_follow, tb_blog

7) GET /blog/of/me, GET /blog/of/user
- 逻辑：按user_id分页查笔记
- 数据库：tb_blog
- Redis：无

-------------------------------------------------
模块：关注
1) PUT /follow/{id}/{isFollow}
- 调用：FollowServiceImpl.follow(followUserId,isFollow)
- 逻辑：关注则写tb_follow并写Redis Set；取关则删tb_follow并删Set成员
- Redis：
  key=follows:{userId}，类型=Set(member=followUserId)
- 数据库：
  表tb_follow

2) GET /follow/or/not/{id}
- 逻辑：查tb_follow是否存在关系
- 数据库：tb_follow
- Redis：无

3) GET /follow/common/{id}
- 调用：FollowServiceImpl.followCommons(id)
- 逻辑：先确保双方关注Set已预热，再做Set交集得到共同关注用户ID，查用户信息返回
- Redis：
  key=follows:{userId}，类型=Set
- 数据库：
  表tb_follow, tb_user

-------------------------------------------------
模块：优惠券与秒杀
1) POST /voucher
2) POST /voucher/seckill
- 调用：VoucherServiceImpl.addSeckillVoucher(voucher)
- 逻辑：保存tb_voucher + 保存tb_seckill_voucher + 秒杀库存写Redis
- Redis：
  key=seckill:stock:{voucherId}，类型=String(value=库存)
- 数据库：
  表tb_voucher, tb_seckill_voucher

3) GET /voucher/list/{shopId}
- 逻辑：按店铺查询优惠券列表
- 数据库：tb_voucher（及关联秒杀信息查询）
- Redis：无

4) POST /voucher-order/seckill/{id}
- 调用：VoucherOrderServiceImpl.seckillVoucher(voucherId)
- 逻辑：
  执行Lua原子校验（库存、一人一单、扣库存、写下单标记）
  通过RabbitMQ异步下单
  消费者拿到消息后加分布式锁并落库
- 关键代码：
  src\main\java\com\hmdp\service\impl\VoucherOrderServiceImpl.java
  src\main\resources\seckill.lua
  src\main\java\com\hmdp\listener\SeckillVoucherListener.java
  src\main\java\com\hmdp\config\RabbitMQConfig.java
- Redis：
  key=seckill:stock:{voucherId}，类型=String（库存）
  key=seckill:order:{voucherId}，类型=Set（已下单用户ID）
  key=lock:order:{userId}，类型=Redisson分布式锁键
  key=icr:order:yyyy:MM:dd，类型=String自增（订单ID序列）
  stream=stream.orders，类型=Stream（Lua里XADD写入，当前Java主链路主要走RabbitMQ）
- 数据库：
  表tb_seckill_voucher（扣库存）
  表tb_voucher_order（保存订单）

三、数据库表结构（项目已定义/使用）
=================================
SQL文件：
dianping-nginx-1.18.0\src\main\resources\db\hmdp.sql

table tb_blog
- 用途：探店笔记主表
- 关键字段：id, shop_id, user_id, title, images, content, liked, comments, create_time, update_time

table tb_blog_comments
- 用途：评论与回复
- 关键字段：id, user_id, blog_id, parent_id, answer_id, content, liked, status, create_time, update_time

table tb_follow
- 用途：用户关注关系
- 关键字段：id, user_id, follow_user_id, create_time

table tb_seckill_voucher
- 用途：秒杀券扩展信息（与voucher 1:1）
- 关键字段：voucher_id(主键), stock, begin_time, end_time, create_time, update_time

table tb_shop
- 用途：店铺主表
- 关键字段：id, name, type_id, images, area, address, x, y, avg_price, sold, comments, score, open_hours, create_time, update_time

table tb_shop_type
- 用途：店铺类型
- 关键字段：id, name, icon, sort, create_time, update_time

table tb_sign
- 用途：签到记录表（当前代码主流程未使用）
- 关键字段：id, user_id, year, month, date, is_backup

table tb_user
- 用途：用户基础信息
- 关键字段：id, phone(唯一), password, nick_name, icon, create_time, update_time

table tb_user_info
- 用途：用户扩展档案
- 关键字段：user_id(主键), city, introduce, fans, followee, gender, birthday, credits, level, create_time, update_time

table tb_voucher
- 用途：优惠券主表
- 关键字段：id, shop_id, title, sub_title, rules, pay_value, actual_value, type, status, create_time, update_time

table tb_voucher_order
- 用途：优惠券订单
- 关键字段：id, user_id, voucher_id, pay_type, status, create_time, pay_time, use_time, refund_time, update_time

四、Redis结构总表（按业务）
=================================
1) 登录验证码
- key: login:code:{phone}
- type: String
- value: 6位验证码
- ttl: 2分钟

2) 登录会话
- key: login:token:{token}
- type: Hash
- value: UserDTO字段映射
- ttl: 30分钟（请求中续期）

3) 商铺缓存
- key: cache:shop:{shopId}
- type: String(JSON)
- value: RedisData{expireTime,data}
- ttl: 逻辑过期（对象内expireTime）

4) 商铺缓存重建锁
- key: lock:shop:{shopId}
- type: String
- value: 1
- ttl: 10秒左右（代码中setIfAbsent 10s）

5) 店铺地理索引
- key: shop:geo:{typeId}
- type: GEO（底层ZSet）
- member: shopId
- score/附加：经纬度位置，查询时返回distance

6) 店铺类型缓存
- key: shop:type:list
- type: ZSet
- member: shopTypeId
- score: sort
- ttl: 1小时

7) 博客点赞
- key: blog:liked:{blogId}
- type: ZSet
- member: userId
- score: 点赞时间戳

8) 关注收件箱（推模式）
- key: feed:{userId}
- type: ZSet
- member: blogId
- score: 推送时间戳

9) 用户关注集合
- key: follows:{userId}
- type: Set
- member: followUserId

10) 秒杀库存
- key: seckill:stock:{voucherId}
- type: String
- value: stock数字

11) 秒杀一人一单集合
- key: seckill:order:{voucherId}
- type: Set
- member: userId

12) 分布式订单锁
- key: lock:order:{userId}
- type: Lock key（Redisson管理）

13) 全局ID递增计数器
- key: icr:{prefix}:{yyyy:MM:dd}
- type: String(自增计数)
- 示例: icr:order:2026:03:24

14) 用户签到位图
- key: sign:{userId}:{yyyyMM}
- type: Bitmap（String按位）
- value: 每日签到位

15) Lua写入流（补充）
- stream: stream.orders
- type: Stream
- fields: userId, voucherId, id
- 备注：当前Java主消费链路主要使用RabbitMQ队列QA/QD

六、给Cursor/另一个AI的优化提示词（可直接复制）
=================================
你现在是本项目的改造AI，请严格基于以下约束执行：
1) 先读功能映射：每个接口对应controller/service/redis/table，禁止猜测不存在的表和键。
2) 新增功能必须给出：
   - 接口定义（URL、method、请求参数、返回结构）
   - 代码落点（Controller/Service/Mapper/前端页面）
   - Redis键设计（key模式、类型、TTL、并发策略）
   - 数据库变更（新表或字段，索引建议，回填策略）
   - 与现有逻辑的兼容点（鉴权、缓存一致性、异常回滚）
3) 涉及高并发场景时，优先说明原子性方案（Lua/事务/幂等/锁）再给代码。
4) 输出格式固定为：
   A. 改造目标
   B. 影响范围文件清单
   C. 后端实现步骤
   D. 前端实现步骤
   E. Redis设计
   F. MySQL设计
   G. 风险与回滚
   H. 验证清单（接口测试+并发测试+缓存命中率）
5) 禁止一次性大改，先给最小可用版本(MVP)再给增强版。

针对你要做的三个AI方向，建议优先级：
P1 店铺点评智能总结（低风险、可快速展示）
P2 点评AI助手（5km推荐，体验亮点最强）
P3 点评质检与风控（价值高但流程复杂）

三个方向统一要求：
- 模型只做“理解与生成”，业务数据检索必须来自本项目数据库和Redis，不可让模型直连数据库。
- 先产出结构化中间结果（JSON）再渲染文案，便于追踪与降本。
- 所有AI结果都要缓存，并设计失效策略与兜底文案。
